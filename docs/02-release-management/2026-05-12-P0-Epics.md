Based on P0 in feature-roadmap.md, I’d plan 8 epics and about 22 implementation slices to completely deliver P0.

  P0 has 20 product capabilities, but several should be grouped together.

  | # | Epic | P0 capabilities covered | Suggested slices |
  | --- | --- | --- | --- |
  | 1 | Tenant, Auth, Access Foundation | Multi-tenant account model, authentication, roles/permissions, user profile/
  preferences | 4 slices |
  | 2 | App Shell Foundation | Workspace shell, workspace detail layout | 2 slices |
  | 3 | Workflow UI Primitives | Right panel, data table, form validation, status badge/workflow state, activity
  timeline primitive | 5 slices |
  | 4 | Audit And API Contract Foundation | Audit event infrastructure, API error contract, OpenAPI/generated client
  workflow | 3 slices |
  | 5 | File Attachment Baseline | Secure upload, download, preview, tenant-scoped file metadata | 2 slices |
  | 6 | Search And Command Foundation | Command palette, global search | 2 slices |
  | 7 | Notification Foundation | Notification center, notification preferences/read state/deep links | 2 slices |
  | 8 | Local Demo And System Readiness | Seed/demo data, environment/system status surface | 2 slices |

  Total: 8 epics, 22 slices.

  A sensible order:

  1. Tenant, Auth, Access Foundation
  2. Audit And API Contract Foundation
  3. App Shell Foundation
  4. Workflow UI Primitives
  5. File Attachment Baseline
  6. Search And Command Foundation
  7. Notification Foundation
  8. Local Demo And System Readiness

  Some can run in parallel after dependencies mature:

  - App Shell and Workflow UI Primitives can overlap after auth/tenant basics exist.
  - File Attachment Baseline can run parallel with Search/Command once tenancy and API error contracts are stable.
  - Seed/Demo and System Status can be done near the end, but seed data can start earlier and evolve.

  I would not run 20 separate runbook cycles. Run the 9-step feature runbook once per slice, so around 22 runbook cycles
  for a complete, polished P0.
