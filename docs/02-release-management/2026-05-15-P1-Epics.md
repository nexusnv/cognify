# P1 Epics - Core Procurement Lifecycle

Based on `docs/01-product/feature-roadmap.md`, P1 should be planned as 8 epics and 33 implementation slices. The goal is to deliver thin, complete procurement lifecycle increments from requisition draft to award handoff without splitting tightly coupled domain behavior across too many separate delivery tracks.

P1 has 35 planned product capabilities. Several should be grouped because they share the same Laravel domain model, frontend workspace, workflow state machine, audit events, and generated API contracts.

| # | Epic | P1 capabilities covered | Suggested slices |
| --- | --- | --- | --- |
| 1 | Requisition Authoring And Intake Foundation | Requisition draft creation, autosave/manual save, department and cost center capture, templates, line item catalog suggestions | 5 slices |
| 2 | Requisition Submission And Workspace | Requisition submission, list/work queue, detail workspace, comments/mentions, change requests, cancellation/withdrawal | 6 slices |
| 3 | Procurement Project Workspace | Procurement project records, project workspace | 2 slices |
| 4 | Approval Orchestration | Approval routing rules, approval tasks, sequential chains, parallel groups, delegation, escalation, SLA tracking, policy preview | 7 slices |
| 5 | Buyer Intake And RFQ Management | Buyer intake review, RFQ creation, vendor invitation to RFQ | 3 slices |
| 6 | Vendor Portal And Quotation Capture | Vendor portal baseline, quotation upload, quotation manual entry, quotation versioning | 4 slices |
| 7 | Quotation Evaluation And Award | Quotation normalization, comparison table, vendor scoring matrix, recommendation and award decision, award approval | 4 slices |
| 8 | Post-Award Handoff And Procurement Calendar | Purchase order request handoff, procurement calendar | 2 slices |

Total: 8 epics, 33 slices.

## Delivery Principles

- Build each epic as a usable workflow slice, not only as CRUD screens.
- Keep requisition, approval, sourcing, quotation, and award state transitions auditable from the first slice.
- Update OpenAPI contracts and regenerate `@cognify/api-client` for every API contract change.
- Use P0 primitives instead of rebuilding workflow UI: workspace detail layout, activity timeline, status badges, data tables, forms, right panels, attachments, notifications, audit events, and tenant-scoped API errors.
- Keep Cognify-specific procurement shells and workflows in `apps/web`; keep Laravel business behavior in `apps/api/Domains/*`.

## Epic 1 - Requisition Authoring And Intake Foundation

**Primary responsibility:** Requisition domain, requester experience, organizational spend metadata.

**Capabilities grouped together:**

- Requisition Draft Creation
- Requisition Autosave and Manual Save
- Department and Cost Center Capture
- Requisition Templates
- Line Item Catalog Suggestions

**Implementation slices:**

1. Requisition draft API and data model: tenant-scoped requisitions, line items, estimated pricing, needed-by date, justification, department, project, and cost center fields.
2. Requester draft form: create/edit draft requisitions with line item editing, validation summaries, save states, and generated client usage.
3. Autosave/manual save reliability: debounce autosave, explicit save action, conflict handling, unsaved-change guard, and audit-visible draft updates.
4. Template application: seed common templates and allow requesters to start or update drafts from template defaults.
5. Line item suggestions: suggest units, categories, common item names, and estimated pricing from seeded catalog-like data without introducing a full catalog domain yet.

**Exit criteria:**

- A requester can create, save, leave, reopen, and continue a draft without data loss.
- Department and cost center fields are validated server-side and exposed through generated clients.
- Draft events are visible in the activity timeline.

## Epic 2 - Requisition Submission And Workspace

**Primary responsibility:** Requisition lifecycle, collaboration, operational queues.

**Capabilities grouped together:**

- Requisition Submission
- Requisition List and Work Queue
- Requisition Detail Workspace
- Requisition Comments and Mentions
- Requisition Change Requests
- Requisition Cancellation and Withdrawal

**Implementation slices:**

1. Submission workflow: required-field validation, draft-to-submitted transition, requester edit locking, audit event, and notification hooks.
2. Requisition work queue: requester and buyer filters for status, requester, owner, department, needed-by date, amount, and updated date.
3. Detail workspace: overview, line items, justification, checklist, activity, approvals placeholder, quotation readiness, and evidence section.
4. Comments and mentions: tenant-scoped discussion threads, mention parsing, notifications, and activity timeline entries.
5. Change requests: buyer/approver return flow with reason, requested fields, requester correction, and resubmission.
6. Cancellation and withdrawal: requester withdrawal and admin cancellation with reason capture, permissions, immutable audit events, and terminal states.

**Exit criteria:**

- Submitted requisitions become the buyer/approver source of truth.
- Collaboration and change requests happen on the record instead of outside the system.
- Terminal and returned states cannot bypass audit or permission checks.

## Epic 3 - Procurement Project Workspace

**Primary responsibility:** Project-level grouping for larger procurement initiatives.

**Capabilities grouped together:**

- Procurement Project Records
- Project Workspace

**Implementation slices:**

1. Project record foundation: tenant-scoped project model, charter, budget, owner, status, related requisitions, and generated API endpoints.
2. Project workspace: project dashboard with budget summary, requisition pipeline, approvals, activity, risks placeholder, and related awards placeholder.

**Exit criteria:**

- Buyers can group related requisitions under a project.
- Project-level activity and ownership are visible before sourcing and awards are added.

## Epic 4 - Approval Orchestration

**Primary responsibility:** Approval policy engine, approval task lifecycle, SLA governance.

**Capabilities grouped together:**

- Approval Routing Rules
- Approval Tasks
- Sequential Approval Chains
- Parallel Approval Groups
- Approval Delegation
- Approval Escalation
- Approval SLA Tracking
- Approval Policy Preview

**Implementation slices:**

1. Routing rule model and versioning: rules by amount, department, category, cost center, project, requester, risk, or vendor.
2. Policy preview: requester and buyer preview of the approval path before submission or routing.
3. Approval task lifecycle: approve, reject, request changes, delegate, and comment actions with permissions and activity events.
4. Sequential chains: ordered approval stages with stage advancement and blocked-stage visibility.
5. Parallel groups: concurrent stakeholder approval groups with completion rules and timeline visibility.
6. Delegation: temporary or standing delegation within policy limits, with assignee visibility and audit trail.
7. Escalation and SLA tracking: due dates, overdue status, fallback approvers, aging metrics, and notification hooks.

**Exit criteria:**

- A submitted requisition can be routed through configured approval paths.
- Approvers can act from a queue and from the requisition detail workspace.
- SLA and escalation state is visible without manual spreadsheet tracking.

## Epic 5 - Buyer Intake And RFQ Management

**Primary responsibility:** Buyer sourcing intake and RFQ setup.

**Capabilities grouped together:**

- Buyer Intake Review
- RFQ Creation
- Vendor Invitation to RFQ

**Implementation slices:**

1. Buyer intake queue: submitted/approved requisitions requiring sourcing review, classification, completeness checks, and sourcing path decision.
2. RFQ creation: create RFQ from requisition or project with scope, line items, due date, required documents, and response instructions.
3. Vendor invitations: invite vendors, track invitation status, response deadlines, communication history, and activity events.

**Slice planning note:** These slices should be delivered as separate runbook cycles unless a later plan deliberately combines adjacent work. A first implementation-ready design may cover only slice 1, but it must preserve explicit downstream boundaries for slice 2 RFQ creation and slice 3 vendor invitations so those capabilities are not treated as dropped scope.

**Exit criteria:**

- Buyers can turn an approved or intake-ready requisition into an RFQ.
- Vendor invitation status and deadlines are tracked from the RFQ workspace.

## Epic 6 - Vendor Portal And Quotation Capture

**Primary responsibility:** External vendor response surface and quotation intake.

**Capabilities grouped together:**

- Vendor Portal Baseline
- Quotation Upload
- Quotation Manual Entry
- Quotation Versioning

**Implementation slices:**

1. Vendor portal baseline: secure external RFQ access, tenant isolation, invitation-token or account-based access, and audit events.
2. Quotation upload: buyer/vendor file uploads linked to RFQs, requisitions, and evidence records.
3. Manual quotation entry: structured quote capture for vendors responding outside the portal or with incomplete documents.
4. Quotation versioning: revised quotations, prior-version retention, current-version markers, and evaluation-time snapshot support.

**Exit criteria:**

- Vendors can view invited RFQs and submit quotation evidence.
- Buyers can manually capture quote data when vendor adoption is incomplete.
- Revised quotations do not overwrite prior evaluated terms.

## Epic 7 - Quotation Evaluation And Award

**Primary responsibility:** Buyer evaluation workspace, recommendation, and award governance.

**Capabilities grouped together:**

- Quotation Normalization
- Quotation Comparison Table
- Vendor Scoring Matrix
- Recommendation and Award Decision
- Award Approval

**Implementation slices:**

1. Quotation normalization: comparable fields for item, quantity, unit price, currency, tax, freight, discount, payment terms, warranty, lead time, and compliance notes.
2. Comparison table: side-by-side vendor comparison by price, delivery, terms, compliance, risk placeholder, and buyer notes.
3. Scoring matrix: configurable criteria for cost, delivery, quality, compliance, risk, sustainability, and past performance.
4. Award decision and approval: selected vendor, rationale, supporting evidence, award approval routing, and auditable decision record.

**Exit criteria:**

- Buyers can evaluate quotes inside Cognify without exporting to spreadsheets.
- Award recommendations preserve rationale, evidence, scores, and approval state.

## Epic 8 - Post-Award Handoff And Procurement Calendar

**Primary responsibility:** Operational handoff after award and time-based procurement visibility.

**Capabilities grouped together:**

- Purchase Order Request Handoff
- Procurement Calendar

**Implementation slices:**

1. Purchase order request handoff: structured ERP/finance handoff package with requisition, award, vendor, line item, pricing, and approval summary.
2. Procurement calendar: RFQ deadlines, approval due dates, expected delivery dates, contract renewal dates, and expiring vendor document placeholders.

**Exit criteria:**

- Awarded procurement work has a clear next operational step even before ERP integration exists.
- Buyers and managers can see time-sensitive procurement events in one calendar surface.

## Suggested Build Order

1. Requisition Authoring And Intake Foundation
2. Requisition Submission And Workspace
3. Approval Orchestration
4. Procurement Project Workspace
5. Buyer Intake And RFQ Management
6. Vendor Portal And Quotation Capture
7. Quotation Evaluation And Award
8. Post-Award Handoff And Procurement Calendar

## Parallelization Guidance

- Requisition Authoring should start first because most P1 capabilities depend on the requisition record and line items.
- Approval Orchestration can begin after the submission state model and department/cost center fields are stable.
- Procurement Project Workspace can run in parallel with Approval Orchestration once requisition-project linking is defined.
- Buyer Intake and RFQ Management can start after submitted or approved requisitions exist; it should not wait for every approval edge case.
- Vendor Portal and Quotation Capture can run in parallel with later RFQ work once RFQ invitation contracts stabilize.
- Quotation Evaluation and Award depends on quotation versioning and normalized quote fields.
- Procurement Calendar can start late, but the event model should be considered while implementing approvals, RFQs, quotations, and awards.

## Recommended Runbook Cycles

Do not run 35 separate runbook cycles. Run the feature development runbook once per implementation slice, so roughly 33 runbook cycles for a complete P1. If delivery needs to be compressed, combine only adjacent slices from the same epic that share the same domain model and UI workspace.

Suggested minimum shippable milestones:

1. Requester can draft, save, submit, revise, and withdraw requisitions.
2. Approvers can route, preview, approve, reject, request changes, delegate, and track overdue approvals.
3. Buyers can intake approved requisitions, create RFQs, invite vendors, and capture quotations.
4. Buyers can compare quotations, score vendors, recommend an award, route award approval, and produce a PO handoff.
5. Managers can monitor procurement deadlines through the calendar.
