# Cognify Desktop Design System Design

## Summary

Cognify needs a new app-level design system for a dense desktop procure-to-pay SaaS. The current web app uses factory shadcn primitives, but page composition still varies by feature. This design defines the next source of truth for Cognify's shell, navigation, page templates, and route composition.

The system uses uncustomized shadcn components and the default shadcn theme. Cognify-specific design work belongs in how the app composes those primitives, not in rewriting or styling primitive components. The target shell pattern is shadcn's official `sidebar-07` block: a primary app rail, an optional contextual secondary sidebar, and a content header inside the workspace.

## Goals

- Establish a Cognify-owned desktop design system for consistent app composition.
- Use uncustomized shadcn components.
- Use the default shadcn theme generated by the current shadcn configuration.
- Keep `packages/ui` as a factory-default shadcn primitive package only.
- Define a new desktop app shell modeled on shadcn `sidebar-07`.
- Support one or two left sidebars depending on route context.
- Make dense data workflows the default design target.
- Define consistent page templates for dashboards, module landing pages, work queues, record detail pages, and form workspaces.
- Give future P2P slices one layout model for purchase orders, receiving, invoices, matching, payment readiness, and operational queues.

## Non-Goals

- No mobile web support in this design system.
- No mobile drawer or responsive sidebar behavior.
- No companion mobile app design in this spec.
- No new color palette, typography override, or custom shadcn theme.
- No custom primitive variants in `packages/ui`.
- No backend/API contract changes.
- No feature workflow redesign beyond layout and navigation composition.
- No implementation in this spec.

## Product Positioning

Cognify web is a desktop operational console for procure-to-pay teams. It should support long work sessions, high-density queues, record review, audit tracing, financial state comparison, and frequent context switching.

The web product is intentionally not optimized for handheld use. Mobile users will be served by a separate companion mobile app plan focused on smaller workflows such as approvals, notifications, receipt confirmation, and quick review. The web app can document a minimum supported viewport, but it should not spend this design-system pass solving mobile layout.

## Foundation Rules

### Shadcn Primitives

`packages/ui` remains the shadcn primitive package.

- Do not customize generated shadcn components for Cognify visual identity.
- Do not place Cognify shell behavior, product copy, navigation rules, procurement state, or P2P variants in `packages/ui`.
- Keep current shadcn registry configuration and default theme unless a separate approved spec changes them.
- Use shadcn components directly in app shell and feature composition.

Expected primitive families:

- Shell: `Sidebar`, `Breadcrumb`, `DropdownMenu`, `Command`, `Button`, `Separator`, `Tooltip`.
- Content: `Card`, `Tabs`, `Table`, `Badge`, `Accordion`, `ScrollArea`, `Popover`.
- Forms: `Form`, `Field`, `Input`, `Textarea`, `Select`, `Checkbox`, `RadioGroup`, `Switch`.
- Feedback: `Alert`, `AlertDialog`, `Skeleton`, `Spinner`, `Empty`, `Sonner`, `Progress`.

### Cognify Composition Layer

The design system composition layer belongs in `apps/web`.

Recommended location:

```txt
apps/web/components/design-system/
  shell/
  navigation/
  page/
  data-display/
  feedback/
```

If implementation prefers preserving `apps/web/components/shell`, that is acceptable, but new shell behavior should still be documented as Cognify design-system composition rather than primitive customization.

## Desktop Shell Model

The authenticated desktop shell has these regions:

```txt
+---------------------------------------------------------------------+
| Primary sidebar | Secondary sidebar | Header                         |
|                 |                   |--------------------------------|
| Product areas   | Context nav       | Page content                    |
|                 |                   |                                |
|                 |                   |                                |
|                 |                   |--------------------------------|
|                 |                   | Footer / status                 |
+---------------------------------------------------------------------+
```

### Primary Sidebar

The primary sidebar represents top-level product areas.

Candidate areas:

- Home
- My Work
- Procurement
- Vendors
- Finance
- Evidence
- Analytics
- Governance
- Admin
- Integrations

The primary sidebar should use shadcn `Sidebar` primitives and the `sidebar-07` visual rhythm. It should not render the complete product inventory. It should help users switch modules.

### Secondary Sidebar

The secondary sidebar is optional. It represents contextual navigation for the selected product area or workspace family.

Examples:

- Procurement: Requisitions, Projects, Buyer intake, RFQs, Quotations, Awards, Purchase orders, Receiving, Calendar.
- Finance: Invoice review, Match exceptions, Invoice approvals, Payment readiness, Payment status, Commitments.
- Governance: Policies, Approval policies, Exceptions, Audit, Risk alerts.
- Admin: Users, Roles, Organization, Notification rules, Integrations, System.

The secondary sidebar should use the same uncustomized shadcn `Sidebar` primitives. It may contain section groups, saved views, and route-aware links.

## Sidebar State Rules

### Routes With Secondary Sidebar

When the route declares a secondary sidebar:

- Render both left sidebars.
- Force the primary sidebar into icon-only collapsed mode.
- Hide or disable primary-sidebar collapse controls.
- The visible collapse toggle controls only the secondary sidebar.
- Secondary sidebar defaults expanded unless the user previously collapsed it.
- Secondary sidebar can collapse to icon mode.
- Primary sidebar remains usable as an icon rail for module switching.

### Routes Without Secondary Sidebar

When the route does not declare a secondary sidebar:

- Render only the primary sidebar.
- Primary sidebar defaults expanded.
- The visible collapse toggle controls the primary sidebar.
- Primary sidebar can collapse to icon mode.
- No secondary sidebar is mounted.

### State Ownership

Sidebar state should be route-context aware.

- `primaryOpen` applies only when no secondary sidebar exists.
- `secondaryOpen` applies only when a secondary sidebar exists.
- A route with secondary navigation should not inherit an expanded primary sidebar from a previous no-secondary route.
- A route without secondary navigation should not inherit secondary sidebar state.
- Persisting collapse state is allowed, but the forced primary-icon rule must always win when a secondary sidebar is present.

## Header

The header belongs inside the content column, not above both sidebars.

Header content:

- Breadcrumbs.
- Current tenant/workspace context.
- Command/search trigger.
- Notifications.
- Theme toggle.
- Account menu.
- Optional route command bar slot for primary page actions.

The header should use shadcn primitives and default theme tokens. It should be visually quiet and optimized for repeated use.

## Footer

The footer should remain quiet and functional.

Possible footer content:

- Workspace label.
- System readiness indicator when permitted.
- Environment/version metadata.
- Background job or sync status when implemented.

Footer content must not become a marketing or decoration area.

## Page Templates

The design system should standardize page structure without forcing every feature into identical content.

### Dashboard

Use for `/dashboard` and operational home surfaces.

Required structure:

- Compact page header.
- High-priority work summary.
- Work queues or status panels.
- Calendar or due-soon panel.
- No marketing hero.

### Module Landing

Use for module roots such as `/procurement`, `/finance`, `/vendors`, `/governance`, and `/analytics`.

Required structure:

- Module title and short operational description.
- Primary actions.
- Saved views.
- Recent records.
- Module-specific queues or health indicators.

### Work Queue

Use for high-frequency operational lists.

Required structure:

- Page header with primary action.
- Filter/search bar.
- Saved view selector when applicable.
- Dense table using shadcn `Table` or approved TanStack Table composition.
- Empty, loading, error, and permission-denied states.
- Bulk actions only when permissions and audit behavior are defined.

### Record Detail

Use for requisitions, RFQs, quotations, awards, purchase orders, receipts, invoices, vendors, and projects.

Required structure:

- Record header with status, identity, metadata, and primary actions.
- Local tabs or sections.
- Main content region.
- Contextual side panel or right panel for evidence, activity, approval, or matching details.
- Activity/audit trail where important state transitions occur.

### Form Workspace

Use for creation/editing workflows.

Required structure:

- Form title and current state.
- Clear save/submit/cancel actions.
- Inline validation.
- Server validation mapping.
- Unsaved-change behavior when applicable.
- Summary panels for totals, policy impact, or approval preview when applicable.

## Dense Data Standards

Cognify pages should optimize for dense operational work.

- Prefer tables, queues, tabs, split panes, and record-local navigation over decorative cards.
- Cards are acceptable for grouped summaries, repeated items, and bounded panels, but not as the default page section wrapper.
- Keep headings compact inside operational views.
- Use tabular numbers for money, quantities, percentages, and due dates where possible.
- Preserve horizontal space for P2P data such as PO lines, receipt quantities, invoice variances, matching status, payment terms, tax, and currency.
- Avoid route-specific raw color palettes. Use shadcn semantic tokens and component variants.
- Use badges with text labels for status; do not rely on color alone.

## Navigation Data Model

Navigation should be a typed app-owned registry.

Each route family can define:

- `primaryArea`: top-level primary sidebar item.
- `secondaryGroups`: optional contextual navigation groups.
- `breadcrumbs`: route-aware breadcrumb builder.
- `pageTemplate`: dashboard, module landing, work queue, record detail, or form workspace.
- `permissions`: visibility rules.
- `implemented`: whether link is usable.

Primary and secondary navigation should be permission-aware. Future roadmap items should not render as active links until routes exist.

## Implementation Architecture

Recommended components:

```txt
apps/web/components/design-system/shell/
  desktop-app-shell.tsx
  primary-sidebar.tsx
  secondary-sidebar.tsx
  shell-header.tsx
  shell-footer.tsx
  shell-sidebar-toggle.tsx

apps/web/components/design-system/navigation/
  navigation-registry.ts
  navigation-types.ts
  route-context.ts
  route-context.test.ts

apps/web/components/design-system/page/
  page-header.tsx
  module-landing-layout.tsx
  work-queue-layout.tsx
  record-detail-layout.tsx
  form-workspace-layout.tsx
```

Implementation may migrate existing shell files instead of creating this exact tree, but the resulting ownership should remain clear:

- `packages/ui`: shadcn primitives only.
- `apps/web/components/design-system`: Cognify composition patterns.
- `apps/web/features/*`: feature workflows, data hooks, forms, and domain-specific content.

## Migration Strategy

1. Create the desktop shell and navigation registry without changing feature behavior.
2. Replace the current single-sidebar shell with the new route-context-aware shell.
3. Add secondary sidebars first for high-density modules: Procurement, Finance, Governance, Admin.
4. Migrate existing routes to page templates in small groups.
5. Keep feature APIs, hooks, generated clients, permissions, and workflow state unchanged.
6. Add tests for shell state behavior before broad page migrations.

Recommended first implementation slice:

- Desktop shell with primary sidebar only.
- Route-context support for optional secondary sidebar.
- One secondary sidebar example for Procurement.
- Dashboard and requisitions list rendered through the new shell.
- Tests proving the two sidebar state modes.

## Testing Requirements

Focused tests should prove:

- Routes without secondary sidebar render expanded primary sidebar by default.
- Routes without secondary sidebar allow primary sidebar icon collapse.
- Routes with secondary sidebar force primary sidebar to icon mode.
- Routes with secondary sidebar allow only secondary sidebar collapse toggling.
- Secondary navigation visibility follows permissions.
- Breadcrumbs and account/header actions still render.
- `packages/ui` remains free of Cognify product shell behavior.

Broader verification after implementation should include:

- `pnpm audit:shadcn-factory-ui`
- `pnpm --filter @cognify/ui typecheck`
- `pnpm --filter @cognify/web typecheck`
- Focused web shell tests
- Representative feature workflow tests for migrated routes

## Open Decisions For Implementation Planning

- Minimum supported desktop viewport width. Recommended starting point: 1280px.
- Whether to preserve the current bare `d` theme shortcut during shell rewrite.
- Whether to migrate existing shell files in place or create `components/design-system` and then remove old shell files.
- Whether secondary sidebar state should persist globally per module or per route family.

## Acceptance Criteria

This design is ready for implementation planning when:

- Desktop-only web scope is accepted.
- Default shadcn theme and uncustomized primitive constraints are accepted.
- `sidebar-07` is accepted as the shell reference pattern.
- Two-sidebar state rules are accepted.
- The first implementation slice is scoped to shell/navigation foundation, not every page migration.
